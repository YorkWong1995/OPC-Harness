# OPC 内部技术分享文档

> 面向公司内部分享，重点说明这个项目做了什么、使用了哪些技术、已采用哪些优化方式，以及后续计划优化方向。

## 1. 分享目标

这次分享的目标不是介绍一个“会写代码的 AI 工具”，而是介绍一个围绕 AI Agent 软件交付流程搭建的工程化框架：**OPC**。

OPC 的核心思想是：把一次软件开发任务拆成产品、实现、验证、复盘等角色环节，并通过工作流、工具、知识检索、状态记录和验收机制，让 AI 参与的软件交付过程更可控、更可追踪、更容易复用。

## 2. 项目做了什么

OPC 当前主要完成了以下几类能力：

1. **定义单人软件公司的角色协作模型**
   - 将 PM、Engineer、QA 作为最小闭环角色。
   - 保留 Architect、Ops、Growth、CEO 等动态角色入口。
   - 每个角色都有明确职责、输入、输出和禁止事项。

2. **实现 PM → Engineer → QA 的最小交付闭环**
   - PM 负责把用户目标结构化为需求、范围和验收标准。
   - Engineer 基于需求和代码上下文完成实现。
   - QA 根据验收标准检查结果，支持通过、退回或人工介入。
   - QA fail 后可以带 defects 回流 Engineer，避免只在末端描述失败。

3. **建立结构化输出协议和 schema 校验**
   - PM、Engineer、QA 的输出必须先给出可解析 JSON。
   - 工作流在流转前解析角色输出，避免纯自然语言导致状态不可控。
   - 失败原因、阻塞项、下一步建议都被纳入结构化字段。

4. **实现运行记录、trace 和指标沉淀**
   - 每次工作流执行生成 `run_id`。
   - 记录阶段开始、阶段完成、工具调用、成本警告等事件。
   - 输出 `run_trace.json`、`run_metrics.json`、`.opc_state.json` 等运行产物。

5. **实现本地知识检索能力**
   - 支持对代码和文档建立索引。
   - 使用向量检索 + BM25 的混合检索方式。
   - 通过 RRF 融合召回结果，兼顾语义问题和精确关键词问题。

6. **实现代码影响分析和 Context Pack**
   - 基于 import graph、符号索引和测试文件关联分析变更影响面。
   - 将相关文件、可能调用方、关联测试、风险点和验证命令注入 Context Pack。
   - 用阶段摘要替代历史全文堆叠，降低上下文膨胀和信息污染风险。

7. **实现工具注册、权限分类和审计基础**
   - 工具统一注册为 name、description、input_schema、output_schema、permission、side_effect、timeout。
   - 工具调用过程记录输入、耗时、错误和结果摘要。
   - 对危险命令采用轻限制和审计策略，避免 MVP 阶段过度复杂。

## 4. 模块拆解

### 4.1 角色模块

相关文件：

- `src/opc/roles.py`
- `src/opc/schema.py`

#### 4.1.1 多角色协作是什么

**是什么：** 把一次软件交付任务拆成多个职责明确的 Agent 角色，每个角色有独立的 system prompt、输入输出协议和工具权限。当前核心角色是 PM（产品经理）、Engineer（工程师）、QA（验收审查），可选角色包括 Architect、Ops、Growth、CEO。

**为什么：** 单个 Agent 处理完整任务时容易”角色混乱”——一边写需求一边写代码一边验收，导致输出质量不可控、失败原因难定位。拆成多角色后，每个角色只关注自己的职责，输出可以被下游角色检查和退回。

**OPC 的做法：** 每个角色是一个独立的 `Agent` 实例，拥有不同的 system prompt 和工具集。PM 没有文件操作工具（只产出需求），Engineer 有读写文件和执行命令的工具，QA 只有只读工具（只检查不修改）。

#### 4.1.2 结构化输出协议

**是什么：** 要求每个核心角色的输出必须先给出一个可解析的 JSON 对象，字段名和类型固定，工作流在流转前用 Pydantic schema 校验。

**为什么：** 如果角色输出是纯自然语言，工作流无法自动判断”PM 是否定义了验收标准”、”Engineer 是否报告了失败”、”QA 是通过还是退回”。结构化输出让状态流转变成可编程的条件判断，而不是靠人工阅读。

**OPC 的做法：**

- PM 输出必须包含：`background`、`goal`、`scope`、`non_goals`、`acceptance_criteria`、`risks`。
- Engineer 输出必须包含：`changed_files`、`implementation_summary`、`test_result`、`known_limits`、`failure_reason`、`blocked_by`、`suggested_next_step`。
- QA 输出必须包含：`status`（pass/fail）、`checked_items`、`evidence`、`defects`、`next_action`（done/rework/human_intervention）。

工作流在收到角色输出后，先提取 JSON 并用对应的 Pydantic model 校验。校验失败则要求角色重试。

#### 4.1.3 动态角色激活

**是什么：** 可选角色（Architect、Ops、Growth）不默认参与每次任务，而是根据任务内容或用户配置动态启用。

**为什么：** 简单的 bug 修复不需要架构师审查，不涉及部署的任务不需要 Ops 检查。默认启用所有角色会增加 token 消耗和执行时间，且可能引入不必要的流程复杂度。

**OPC 的做法：** 通过关键词匹配推断：任务描述中包含”架构”、”设计方案”、”模块边界”等词时激活 Architect；包含”部署”、”发布”、”上线”时激活 Ops。也支持 CLI 参数显式指定。

#### 4.1.4 计划优化

- 引入更细粒度的角色能力声明，例如可用工具、可写文件范围、可触发状态。
- 为 Architect、Ops、Growth 也补充结构化 JSON schema 校验。
- 让角色选择从关键词推断升级为基于任务类型和上下文的策略判断。

### 4.2 工作流模块

相关文件：

- `src/opc/workflow.py`
- `src/opc/workflow_spec.py`
- `src/opc/run_store.py`
- `src/opc/schema.py`

#### 4.2.1 Harness 工作流是什么

**是什么：** 一个状态机驱动的角色编排引擎。把"需求 → 实现 → 验收"的软件交付过程建模为一系列状态（待澄清 → 已定义 → 实现中 → 待验收 → 已通过/已退回 → 已复盘），每个状态对应一个角色的执行，状态之间的流转由角色输出决定。

**为什么：** 如果只是依次调用 PM、Engineer、QA 三个 Agent，没有状态管理，那么：QA 失败后无法自动回流 Engineer；无法限制返工次数防止死循环；无法从中断处恢复；无法记录每个阶段的耗时和 token。状态机让流程变得可控、可恢复、可观测。

**OPC 的做法：** `HarnessWorkflow` 类维护当前状态、已完成阶段列表和产物路径。每个阶段执行后根据角色输出决定下一步：QA 输出 `status=pass` 则进入"已通过"，`status=fail` 则进入"已退回"并回流 Engineer。超过 `max_rework_attempts` 则终止并进入人工介入。

#### 4.2.2 QA 失败回流机制

**是什么：** QA 判定验收不通过时，把发现的 defects 作为输入传回 Engineer，让 Engineer 针对性修复后再次提交验收。

**为什么：** 传统做法是 QA 失败后整个流程终止，需要人工重新启动。自动回流让 Agent 系统具备自我修正能力，减少人工干预频率。

**OPC 的做法：** QA 输出 `next_action=rework` 时，工作流把 QA 的 `defects` 列表拼接到 Engineer 的下一轮输入中。同时递增 `rework_attempts` 计数器，超过阈值（默认 3 次）后停止回流，进入人工介入状态。

#### 4.2.3 声明式 Workflow Spec

**是什么：** 用数据结构（而非硬编码 if-else）描述状态流转规则。`WorkflowSpec` 定义了状态列表、流转条件和终止状态。

**为什么：** 硬编码的状态机难以扩展和配置。声明式 spec 让用户可以通过配置文件定义自己的工作流，而不需要修改代码。P3 阶段先实现最小 spec（只表达 QA pass/fail 的流转），为后续完整 workflow 引擎预留接口。

**OPC 的做法：** `DEFAULT_WORKFLOW_SPEC` 定义了 7 个状态和 7 条流转规则。`next_state(current, condition)` 方法根据当前状态和条件查找下一个状态。`is_terminal(state)` 判断是否到达终止状态。

#### 4.2.4 Context Pack 与阶段摘要

**是什么：** 多阶段工作流中，下游角色需要了解上游发生了什么。Context Pack 是一个结构化数据包，包含任务目标、验收标准、约束、相关文件、验证命令和风险点。阶段摘要（StageSummary）是每个阶段完成后提炼的关键事实。

**为什么：** 如果把所有历史原文（PM 的完整 PRD、Engineer 的完整实现说明、QA 的完整验收报告）都传给下游角色，会导致：上下文窗口膨胀、重复信息干扰、过期信息污染。用摘要替代全文，让下游角色拿到更短、更稳、更可执行的上下文。

**OPC 的做法：**

- 每个阶段完成后，工作流提取 `StageSummary`（目标、决策、变更文件、验证结果、风险、下一步）。
- 构造 `ContextPack` 时，从所有阶段摘要中聚合相关文件和风险，并调用 `ImpactAnalyzer` 补充依赖分析结果。
- Context Pack 作为下游角色 prompt 的一部分注入，替代历史全文堆叠。
- 每次构造 Context Pack 都记录 `context_pack_created` 事件，方便复盘时追踪"角色看到了什么上下文"。

#### 4.2.5 Append-Only Event Log

**是什么：** 工作流执行过程中的所有关键事件（阶段开始、阶段完成、工具调用、成本警告、QA 退回等）都以追加方式写入 `run_events.jsonl` 文件。

**为什么：** 追加写入保证事件不会被覆盖或丢失，即使工作流中断也能从事件日志恢复状态。同时支持后续的复盘分析、质量统计和 UI 展示。

**OPC 的做法：** `RunStore` 类管理事件日志。每次 `append(event_type, **payload)` 都同时写入内存列表和磁盘文件。`write_trace()` 把完整事件序列、最终状态和指标汇总写入 `run_trace.json`。

#### 4.2.6 计划优化

- 将更多硬编码阶段迁移到声明式 workflow spec。
- 支持子工作流，例如 Engineer 内部的 code → test → lint。
- 支持并行独立任务，但保持主链路同步可控。
- 扩展 Context Pack 的生成策略，例如按角色裁剪字段、按风险等级调整验证建议。
- 提供更完整的中断恢复能力，包括消息、工具结果和阶段产物恢复。

### 4.3 Agent 与工具模块

相关文件：

- `src/opc/agent.py`
- `src/opc/tools/tool_registry.py`

#### 4.3.1 Tool Use 循环是什么

**是什么：** Agent 调用 Claude API 时，模型可能返回"我需要调用一个工具"的请求（tool_use），而不是直接给出文本回答。Agent 执行工具后把结果返回给模型，模型再决定是继续调用工具还是给出最终回答。这个"调用 → 执行 → 返回 → 再调用"的过程就是 tool use 循环。

**为什么：** 软件工程任务不能只靠"想"，需要实际读文件、写代码、执行命令、查询知识库。Tool use 让 Agent 从"纯文本生成器"变成"能操作环境的执行者"。

**OPC 的做法：** `Agent.run(message)` 方法内部维护一个消息列表，循环调用 Claude API 直到 `stop_reason=end_turn`。每次收到 `stop_reason=tool_use` 时，遍历所有 tool_use block，执行对应工具，把结果作为 tool_result 追加到消息列表中，然后继续调用 API。

#### 4.3.2 工具注册协议

**是什么：** 每个工具通过统一的 `ToolDefinition` 数据结构注册，包含 name、description、input_schema、output_schema、permission、side_effect、timeout。

**为什么：** 统一注册协议让工作流可以：按权限过滤工具（QA 只能用 read 权限工具）；按 side_effect 判断是否需要审计；按 timeout 设置执行超时；按 schema 生成 Claude API 的 tools 参数。

**OPC 的做法：** 内置工具（read_file、write_file、edit_file、grep、search_knowledge、git_status 等）在模块加载时注册到全局 `_TOOL_REGISTRY`。项目级插件工具从 `opc_plugins/` 目录动态加载。`list_tool_schemas(permissions={"read"})` 可以只返回只读工具的 schema 列表。

#### 4.3.3 命令安全与审计

**是什么：** `run_command` 工具允许 Agent 执行 shell 命令，但通过白名单和审计机制控制风险。

**为什么：** 不限制的命令执行可能导致 Agent 删除文件、推送代码、安装恶意包。但完全禁止命令执行又会让 Agent 无法运行测试、编译代码、查看 git 状态。需要在能力和安全之间找平衡。

**OPC 的做法：**

- **命令白名单**：只允许 python、pip、npm、git、pytest、gcc、make 等有限命令集合。
- **危险参数识别**：对 `git push --force`、`git reset --hard`、`pip install --pre`、`rm -rf` 等模式进行审计记录。
- **工作目录限制**：命令执行限制在项目 workspace 内，不能访问项目外的文件系统。
- **审计日志**：所有工具调用（包括输入参数、耗时、错误）写入 `tool_audit.jsonl`，支持事后审查。

P3 阶段采用"轻限制 + 重审计"策略：不做人工确认拦截，但所有危险操作都有记录。

#### 4.3.4 API 调用重试与容错

**是什么：** 调用 Claude API 时可能遇到上游服务波动（500、502、503、529）。Agent 内置指数退避重试机制。

**为什么：** 工作流执行可能持续数分钟，中间一次 API 超时不应该导致整个流程失败。自动重试让系统在上游短暂不可用时能自我恢复。

**OPC 的做法：** `_call_with_retry` 方法最多重试 3 次，基础间隔 30 分钟（适配上游限流恢复周期），每次翻倍。只对可重试状态码（500/502/503/529）和超时错误重试，其他错误立即抛出。

#### 4.3.5 计划优化

- 将工具权限从 read/write/execute 扩展为更细粒度策略。
- 对高风险工具增加可配置人工确认节点。
- 给工具输出增加标准错误类型，便于工作流做自动恢复或升级人工介入。
- 将工具调用结果和安全审计进一步接入 UI 或历史记录。

#### 4.4.1 RAG 是什么

RAG（Retrieval-Augmented Generation）是一种让 LLM 在回答问题前先检索相关文档的技术模式。流程是：用户提问 → 从知识库中检索相关片段 → 把检索结果和问题一起送给 LLM → LLM 基于检索到的内容生成答案。

**为什么需要 RAG：** LLM 的训练知识有截止日期，且不了解你的私有项目代码。如果直接问”这个 SDK 的初始化顺序是什么”，LLM 只能靠训练数据猜测，很可能编造不存在的 API。RAG 让 LLM 先”查资料”再回答，把回答建立在真实文档之上。

**OPC 中怎么做：** Agent 在执行任务前，通过 `search_knowledge` 工具查询项目知识库，把检索到的代码片段和文档段落注入到 prompt 中。Engineer 和 QA 角色都可以在实现和验收过程中主动检索项目知识。

#### 4.4.2 为什么用混合检索

单一检索方式无法覆盖软件项目中的所有问题类型：

| 问题类型 | 向量检索效果 | BM25 效果 | 原因 |
| --- | --- | --- | --- |
| “相机怎么连接” | 好 | 一般 | 语义理解，不依赖精确词汇 |
| “MV_CC_StartGrabbing” | 差 | 好 | 精确函数名，向量模型不认识 |
| “-1003 错误码” | 差 | 好 | 数字+符号，向量空间中无语义 |
| “如何处理设备被占用” | 好 | 一般 | 自然语言描述，BM25 需要精确关键词 |

**向量检索**把文本转成高维向量，通过余弦相似度找语义相近的内容。适合模糊问题、同义词、跨语言理解。

**BM25**是经典的词频-逆文档频率算法，按关键词精确匹配打分。适合函数名、错误码、配置项、文件路径等精确定位。

**OPC 的做法：** 每次查询同时走向量检索和 BM25 两条路线，各自返回 top-N 候选结果，再通过 RRF 融合成一个排序列表。这样无论用户问的是模糊语义问题还是精确定位问题，都能找到相关内容。

#### 4.4.3 RRF 融合算法

**是什么：** Reciprocal Rank Fusion（RRF）是一种多路检索结果融合算法。它不依赖各路检索的原始分数（因为不同检索器的分数量纲不同），而是只看每个结果在各路中的排名。

**公式：**

```
RRF_score(d) = Σ 1/(k + rank_i(d))
```

其中 `k` 是平滑常数（OPC 使用 k=60），`rank_i(d)` 是文档 d 在第 i 路检索中的排名。

**为什么用 RRF：**

1. 不需要归一化不同检索器的分数（向量相似度是 0-1，BM25 分数可能是 0-50）。
2. 不需要手工调两路权重。
3. 在两路都排名靠前的结果会获得更高融合分数，自然实现”交叉验证”。
4. 实现简单，效果稳定，是学术界和工业界广泛使用的标准方法。

**OPC 的做法：** 向量检索和 BM25 各取 top-30，按 RRF 公式计算每个 chunk 的融合分数，取 top-10 作为最终结果送给 LLM。

#### 4.4.4 语言感知分块

**是什么：** 把源文件切成适合检索的小片段（chunk）。不同于简单的固定行数切分，语言感知分块会识别代码和文档的结构边界。

**为什么：** 如果在函数中间切断，检索到的片段可能只有半个函数，LLM 无法理解完整逻辑。按结构边界切分能保证每个 chunk 是一个语义完整的单元。

**OPC 的做法：**

- **代码文件**：目标 100 行/chunk，overlap 10 行。Python 在顶层 `def`/`class` 后的空行处切分；C/C++/Java 在列 0 的 `}` 后切分。超大块继续用固定窗口兜底，过小块合并。
- **文档文件**：目标 200 行/chunk，overlap 50 行。优先在 Markdown 标题处切分，超长段落用窗口切分。
- **配置文件**：JSON 按顶层 key 切分，YAML 按顶层 key 切分。

#### 4.4.5 增量索引

**是什么：** 只对发生变更的文件重新分块和入库，而不是每次都全量重建整个索引。

**为什么：** 大型项目可能有数千个文件，全量重建耗时长。日常开发中每次只改几个文件，增量更新可以把索引时间从分钟级降到秒级。

**OPC 的做法：** 索引时为每个文件计算内容 hash 并记录在 `meta.json` 中。下次索引时对比 hash：hash 没变的文件跳过；hash 变了的文件删除旧 chunk、重新分块入库；文件被删除的清理旧 chunk。BM25 索引需要全量重建（算法限制），但 ChromaDB 向量索引支持增量删除和添加。

#### 4.4.6 符号搜索

**是什么：** 基于 Python AST（抽象语法树）解析源代码，提取函数、类、方法的定义信息（名称、签名、行号、所属类），建立符号级索引。

**为什么：** BM25 和向量检索都是文本级的，无法精确回答”这个函数定义在哪里”、”这个类有哪些方法”这类结构化问题。符号搜索提供精确的代码结构定位能力。

**OPC 的做法：**

- 用 `ast.parse` 解析 Python 文件，遍历 AST 节点提取 `FunctionDef`、`AsyncFunctionDef`、`ClassDef`。
- 记录每个符号的名称、类型（function/method/class）、文件路径、行号、完整签名、所属类。
- 支持模糊搜索（`search`）、精确定义定位（`find_definition`）、文件内符号列表（`definitions_in_file`）、类方法查询（`methods_of_class`）。

#### 4.4.7 Import Graph 与依赖分析

**是什么：** 解析 Python 文件中的 `import` 和 `from ... import` 语句，构建文件间的依赖关系图。

**为什么：** 修改一个文件时，需要知道哪些文件依赖它（可能受影响）、它依赖哪些文件（需要理解的上下文）。这些信息对 Engineer 理解变更范围、对 QA 确定测试范围都很关键。

**OPC 的做法：**

- 用 AST 提取每个文件的 import 语句，解析相对导入为绝对模块路径。
- 建立双向查询能力：`file_dependencies_of(file)` 查找文件依赖的模块；`file_dependents_of(file)` 查找依赖该文件的其他文件。
- 索引构建时自动计算依赖图并写入 index metadata 的 `file_dependencies` 字段。
- `impact_analysis(file)` 综合正向和反向依赖，列出修改该文件可能影响的所有项目内文件。

#### 4.4.8 测试文件关联

**是什么：** 给定一个源文件，自动推荐与之相关的测试文件。

**为什么：** 修改代码后需要跑相关测试，但大型项目中测试文件可能分散在不同目录、使用不同命名约定。手动查找容易遗漏。

**OPC 的做法：** 三种策略叠加：

1. **命名约定**：源文件 `config.py` → 查找 `tests/test_config.py` 或 `tests/config_test.py`。
2. **镜像路径**：源文件 `src/opc/workflow.py` → 查找 `tests/opc/test_workflow.py`。
3. **Import 关系**：扫描测试目录中所有 `test_*.py`，检查哪些文件 import 了目标模块。

三种策略的结果去重合并，覆盖率比单一策略高。

#### 4.4.9 轻量影响分析

**是什么：** 给定一组变更文件，综合 import graph、符号索引和测试关联，输出完整的影响评估报告。

**为什么：** 单独看依赖图只知道”谁依赖谁”，单独看测试关联只知道”该跑哪些测试”。影响分析把这些信息整合成一个结构化结果，直接告诉工作流：哪些文件可能受影响、有哪些风险点、建议跑哪些验证命令。

**OPC 的做法：** `ImpactAnalyzer` 接收变更文件列表，执行以下步骤：

1. 对每个变更文件查询 import graph 的依赖方和被依赖方。
2. 对每个变更文件查询符号索引，列出被修改文件中定义的符号。
3. 对每个变更文件查询测试关联，找到相关测试。
4. 汇总输出 `ImpactResult`：target_files、related_files、possible_callers、related_tests、risk_points、validation_commands。

这个结果会被注入到工作流的 Context Pack 中，让 Engineer 和 QA 在执行时就能看到变更的潜在影响。

#### 4.4.10 RAG 防幻觉策略

**是什么：** 即使检索到了正确文档，LLM 仍然可能编造不存在的内容（幻觉）。防幻觉策略是一组 prompt 规则和输出格式约束，用于降低 LLM 编造信息的概率。

**为什么会产生幻觉：**

1. LLM 倾向于”帮忙”——宁可编一个答案也不说不知道。
2. 训练知识干扰——把自研 SDK 和通用库知识混在一起。
3. 检索内容不够明确——文档没说不支持，LLM 就脑补支持。

**OPC 的三层防幻觉策略：**

1. **严格 prompt**：核心规则从”不要编造”改为”没有就说不存在”。禁止用训练知识补充检索结果中没有的内容。
2. **强制标注来源**：每个事实后面必须标注 `[来源: 文件路径:行号]`。无法标注来源的陈述不允许出现。
3. **分离事实和推断**：回答分为”事实（来自文档）”、”推断（文档中未明确说明）”和”结论”三个区块，让用户一眼区分哪些是检索到的、哪些是 AI 猜的。

#### 4.4.11 计划优化

- **Embedding 升级**：当前 MiniLM 是英文模型，对中文语义理解弱。计划升级到 bge-m3（2GB），支持中英混合、8192 token 长文本。
- **Rerank 精排**：RRF 融合后的 top-30 结果中仍有噪音。计划加入 reranker 模型对 top-30 精排，只取 top 5-10 送给 LLM，提升答案准确度。
- **查询类型自动调权**：根据问题内容自动调整向量/BM25 权重。问题包含函数名、错误码、大写+下划线时偏 BM25；纯中文模糊问题偏向量。
- **上下文扩展**：召回 chunk 后自动扩展同章节、邻近 chunk 或同 symbol 的相关内容，避免只看到半个函数或半个表格。
- **影响分析精度增强**：从文件级依赖扩展到函数/类级调用关系，并按变更类型（签名变更 vs 实现变更）给出不同风险等级。
- **结构化表格解析**：把表格拆成表头+每行+整体摘要，转成可检索文本。例如”错误码 -1003 表示设备被占用”，让 BM25 能精确命中。
- **Agentic RAG**：复杂问题允许 LLM 自主决定查几次、换什么关键词再查、结果够不够。最多 3 轮检索，超过后标注”信息可能不完整”。

### 4.5 可观测性与质量评估模块

相关文件：

- `src/opc/workflow.py`
- `src/opc/run_store.py`
- `tests/test_metrics.py`
- `tests/test_quality_metrics.py`

#### 4.5.1 为什么需要可观测性

**是什么：** 可观测性是指系统运行过程中的关键信息可以被记录、查询和分析。在 OPC 中，这意味着每次工作流执行的阶段耗时、token 消耗、工具调用、失败原因和最终结果都可以被追踪。

**为什么：** AI Agent 的执行过程是”黑盒”——你不知道它为什么调了 5 次工具、为什么花了 50000 token、为什么 QA 退回了 3 次。没有可观测性，就无法定位问题、优化成本、评估质量，也无法向团队解释”这次 Agent 到底做了什么”。

#### 4.5.2 阶段级指标

**是什么：** 每个工作流阶段（PM、Architect、Engineer、QA 等）完成后，记录该阶段的 input tokens、output tokens、耗时、工具调用次数和 API 调用次数。

**为什么：** 不同阶段的资源消耗差异很大。Engineer 可能调用大量工具导致 token 消耗高，PM 可能因为需求不清导致多次重试。阶段级指标让你能精确定位”钱花在哪里”和”时间花在哪里”。

**OPC 的做法：** `_record_stage_metrics` 方法在每个阶段执行后从 Agent 实例读取累计 token 和工具调用数，写入 `WorkflowState.stage_logs`。最终汇总为 `run_metrics.json`，包含各阶段明细和总计。

#### 4.5.3 质量指标

**是什么：** 衡量一次工作流执行”做得好不好”的结构化数据，包括 QA 是否通过、返工次数、人工介入次数、失败类型分布。

**为什么：** 单看”最终通过了”不够。如果每次都要返工 3 次才通过，说明 PM 的需求定义或 Engineer 的实现质量有问题。质量指标让你能发现系统性问题并针对性优化。

**OPC 的做法：** `generate_metrics` 函数从 WorkflowState 中提取质量数据，写入 `run_metrics.json` 的 `quality` 字段。包含 `qa_passed`、`rework_attempts`、`human_interventions`、`failure_types` 等。

#### 4.5.4 成本观测

**是什么：** 记录和告警 token 用量和 API 调用频率，但 P3 阶段不做硬阻断。

**为什么：** LLM API 按 token 计费，一次失控的工作流可能消耗大量 token。但过早做硬限制会导致正常任务被中断。先观测、再优化、最后才限制。

**OPC 的做法：** 配置中定义 `workflow_token_limit` 和 `role_token_limit`。每个阶段完成后检查累计 token，超过阈值时打印警告并记录 `cost_warning` 事件，但不中断执行。后续版本再根据观测数据决定是否启用硬限制。

#### 4.5.5 计划优化

- 提供可视化 run timeline，让非技术人员也能理解工作流执行过程。
- 对失败类型做聚合分析，定位高频失败环节和角色。
- 建立任务级质量评分，例如需求清晰度、实现完整度、验证覆盖度。
- 将 trace 与分享文档、复盘文档联动，自动生成项目演进材料。

## 5. 项目中的关键优化思想

### 5.1 从“生成答案”优化为“约束流程”

传统 AI 编程工具容易直接给出代码，但真实软件交付需要需求、实现、验证、回退和复盘。OPC 的优化点是把 AI 放进一个流程框架中，让模型输出变成可检查的中间产物。

### 5.2 从“自然语言输出”优化为“结构化契约”

PM、Engineer、QA 都要求先输出 JSON。这样工作流可以解析状态、判断失败、提取 defects，而不是靠人工读一段自然语言后再决定下一步。

### 5.3 从“单次调用”优化为“可复盘运行”

每次运行都有 run_id、state、events、metrics 和 artifacts。后续可以分析每个阶段的输入输出、耗时、token 和工具调用。

### 5.4 从“纯向量检索”优化为“混合检索”

软件项目里很多问题不是纯语义问题，而是精确定位问题，例如函数名、错误码、配置项、文件路径。混合检索比单一路线更适合代码和文档联合查询。

### 5.5 从“历史全文堆叠”优化为“Context Pack”

多阶段 Agent 工作流如果直接传递所有历史原文，会导致上下文膨胀、重复信息和过期信息污染。OPC 现在用 `StageSummary` 保留阶段级事实，再用 `ContextPack` 组合任务目标、验收标准、相关文件、验证建议和风险点，让下游角色拿到更短、更稳、更可执行的上下文。

### 5.6 从“只看修改文件”优化为“影响面分析”

代码变更不只影响当前文件，还可能影响调用方、依赖方和关联测试。OPC 通过 import graph、符号索引和测试关联，把可能受影响文件、风险点和建议测试命令提前提供给工作流，降低漏测和上下文遗漏风险。

### 5.7 从“过早平台化”优化为“MVP 闭环优先”

项目没有一开始做完整调度器、多租户、权限沙箱或复杂 workflow 引擎，而是先把 PM → Engineer → QA 的最小链路跑通，再逐步扩展。

## 6. 后续优化路线

### 6.1 短期优化

- 完善自定义 workflow spec，减少硬编码状态。
- 增强 run trace 和 Context Pack 展示，方便分享和复盘。
- 给 Architect、Ops、Growth 增加 schema 校验。
- 优化影响分析策略，让风险点、关联文件和测试建议更贴近真实变更。
- 优化工具错误类型，让工作流能更准确区分可重试、不可重试和需人工介入。

### 6.2 中期优化

- 升级 RAG 检索效果：bge-m3、rerank、查询改写、上下文扩展。
- 增强影响分析：函数级调用关系、跨语言依赖、风险分级和验证策略推荐。
- 增加更完整的状态恢复能力。
- 增加工作流质量评分和失败趋势分析。
- 支持更丰富的插件工具和项目级 skill。

### 6.3 长期优化

- 支持多项目、多任务和子工作流。
- 建立更完整的权限隔离和资源调度。
- 将 OPC 从本地单项目 harness 扩展成可持续运行的单人软件公司操作系统雏形。

建议准备三个材料：

如果需要现场演示，优先选择轻量命令：

```bash
opc index --name opc-docs --dirs docs src/opc --extensions .md .py
opc query "PM 输出 schema 包含哪些字段" --name opc-docs --no-llm
```

避免在分享现场跑耗时长或依赖外部 API 的完整工作流。完整工作流可以提前准备 `artifacts/run_trace.json` 和截图。

## 9. 一句话总结

OPC 的核心价值不是让 AI 一次性替人完成所有开发，而是把 AI 放进一个有角色、有契约、有工具、有验证、有记录的工程 harness 中，让软件交付过程变得更可靠、更可复盘、更容易持续优化。
