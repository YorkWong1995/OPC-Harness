# OPC 知识检索模块 — 设计讨论记录

> 记录于 2026-04-28，基于项目开发过程中的问答讨论整理

---

## 1. 项目背景

### 1.1 痛点

- 人员变动快，新人对大量代码无从下手
- 多个相机 SDK 项目，跨 Windows/Linux
- 每个 SDK 自研，编译、平台、代码复杂
- 文件夹多，文档多，AI 也不方便多项目联合查阅
- 提问不一定精确，不能只依赖精确搜索

### 1.2 目标

构建一个知识检索工具：

- 输入 N 个目录/文件（代码 + 文档）
- 模糊或精确提问后精准定位说明位置和解决方案
- 向量检索 + 精确查找多路查询
- 结果可直接运用到问题解决中

### 1.3 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| Embedding | ONNXMiniLM_L6_V2（MVP）→ 后续升级 bge-m3 | MVP 零额外下载；bge-m3 对中文更优 |
| 向量库 | ChromaDB | 本地持久化，API 简单，适合 MVP |
| 精确搜索 | rank-bm25 + jieba | BM25Okapi + 中文分词 |
| 融合策略 | Reciprocal Rank Fusion (RRF, k=60) | 标准多路融合，不用调参 |
| LLM 生成 | 复用现有 Agent + 本地代理 | 已有 172.16.40.227:3000 |

## 2. 向量数据库为什么选 ChromaDB

选择 ChromaDB 的原因：

1. **本地持久化简单** — 索引直接保存在本机，不需要单独启动服务
2. **接入成本低** — Python API 简单，创建 collection → 写入 chunk → 查询 top-k
3. **支持本地 embedding** — 内置 ONNXMiniLM，数据不出内网
4. **适合 MVP 验证** — 能用较少代码支撑检索闭环
5. **后续可替换** — 向量库封装在 vector_store.py 中，后续可换 FAISS/LanceDB/Qdrant

**不是最终不可替换的架构绑定。**

## 3. 混合检索后还需要 Rerank 吗

### 结论：MVP 不加，中长期预留。

- **现在**：向量 + BM25 + RRF 已经能覆盖大部分"不精确提问 + 精确定位代码"的需求
- **Rerank 的价值**：召回结果较多时重新排序，提升 top 结果准确度
- **Rerank 的代价**：增加延迟、增加模型依赖、CPU 机器可能明显变慢

### 建议路线

```
第一阶段：向量检索 + BM25 + RRF，先验证能不能找到正确位置
第二阶段：top 30 召回 → rerank → top 8 给 LLM，提升排序准确度
第三阶段：按问题类型动态决定是否 rerank
```

RAG 实际由两条独立流程组成：**索引时**完成切分入库，**查询时**完成检索与回答。Chunk 属于索引时流程，所以不会出现在查询链路里。

索引时流程（一次性 / 增量，零 LLM 调用）：

```
源目录（代码 + 文档 + 配置）
  → 语言检测 detect_language
  → Chunk 切分（CodeChunker / DocChunker / ConfigChunker，详见 §4）
  → Embedding 写入向量库（ChromaDB）
     + BM25 索引构建（rank-bm25 + jieba）
     + file_dependencies 元数据写入 meta.json
```

查询时理想流程：

```
用户问题
  → query rewrite / 多路查询
  → 向量检索 + BM25 + 精确符号搜索
  → RRF 融合 top 30-50
  → rerank 精排 top 5-10
  → context expansion（邻近 / 文件依赖 / 同章节）
  → LLM 生成答案 + 文件位置
```

## 4. Chunk 策略设计

### 4.1 当前实现（src/opc/knowledge/chunker.py）

实际落地为三类 chunker，自动按语言分发（`detect_language` + `chunk_file`）：

| Chunker | 适用语言 | 目标大小 | Overlap | 切分策略 |
|---------|---------|---------|---------|---------|
| CodeChunker | python / cpp / c / java / csharp / go / rust / javascript / typescript ... | 100 行 | 10 行 | 按语言识别边界 + 兜底窗口 |
| DocChunker | markdown / rst / 其他文本 | 200 行 | 50 行 | 标题边界优先，超长段再窗口切 |
| ConfigChunker | json / yaml | — | — | 按顶层 key 切分；无命中则降级为 DocChunker |

CodeChunker 的边界识别按语言分支：

- **Python**：在顶层 `def` / `class` / `async def` 之后的首个空行处切分。
- **C/C++/Java/C#/Go/Rust/JavaScript/TypeScript**：在列 0 的 `}` 之后的空行处切分（brace_splits）。
- **其他代码**：在空行处切分。
- **兜底**：无切分点时按固定窗口；区块超过 `target_lines * 2` 时继续窗口切分；过小相邻区块由 `_merge_small` 合并。

DocChunker 的 Markdown 处理有两种模式：

- **默认模式 `_markdown_chunks`**：按所有 `#~######` 标题切分，超长段（>1.5×target_lines）再窗口切。
- **H2 模式 `_markdown_h2_chunks`（heading_level=2）**：仅按 `##` 切分，并自动在每个 chunk 头部注入最近的 H1 + 当前 H2 作为标题上下文前缀；对超长段窗口切分时，子块也会沿用同一前缀。这样保证窗口块在嵌入和检索时不丢失章节归属。

RST 通过识别上/下划线标题切分；其他纯文本走 `_window_chunks` 固定窗口。

### 4.2 已落地的检索时上下文扩展（retriever.py）

设计文档原本归在"4.3 更好的方向"里的几项已实现，由 `Retriever.retrieve` 串联：

- **查询改写 `_rewrite_query`**：识别到代码意图（含标识符、`_CODE_QUERY_HINTS` 关键词或代码意图标记词）时，在 query 末尾追加"代码关键词:"扩展词。
- **代码偏置 `_apply_query_bias`**：对 RRF 融合结果按 chunk 语言、路径前缀（`src/`、`tests/`）和标识符命中给定 bonus，使代码 chunk 在代码意图问题中靠前。
- **Context expansion `expand_context`**：召回的每个 chunk 自动加入相邻 chunk 和文件依赖（`file_dependencies`）首块，作为"邻近扩展 / 同模块扩展"，并在结果里标注 `expansion_reason`。

### 4.3 P10 Golden Eval 标准

轻量 RAG eval 使用 `tests/fixtures/rag_eval_dataset.json` 作为 golden query 数据集。每条 query 至少包含：

| 字段 | 含义 |
| --- | --- |
| `question` | 用户问题，覆盖中文自然语言、英文代码符号、文档章节、Qt/插件问题和误召回样例 |
| `relevant_files` | 期望召回文件路径 |
| `relevant_chunks` | 期望行号或 chunk 定位，格式如 `path::Lx-Ly` |

默认本地入口为 `python scripts/run-rag-eval.py --top-k 3`，使用小语料内存 BM25 计算 hit-rate、MRR、NDCG、命中/未命中明细和失败原因。不调用 LLM，不重建大型向量索引。Release gate 以当前 golden eval 基线作为阻塞门槛，向量索引、bge/faiss 或真实项目大索引可作为补验项记录。

### 4.4 当前方案仍存在的不足

当前仍是**结构启发式 chunk**，不是真正的语义边界 chunk：

- 不能判断一段内容是否在讲同一个概念。
- 参数说明、表格、示例代码可能被切散。
- 同一个业务流程的多个函数可能分到不同 chunk。

### 4.5 待落地的改进方向

```
L1 原子 chunk：函数、标题段落、表格、代码块
L2 语义 chunk：围绕一个主题，把相关原子 chunk 合并
L3 上下文包：查询时动态把相邻/相关 chunk 拼成回答上下文（已部分落地：邻近+依赖扩展）
```

仍未实现的改进点：

- 给 Chunk 增加结构化 metadata：heading_path、block_type、symbol_name、parent_id、prev_id、next_id。
- 文档解析改成 block parser（heading、paragraph、table、code_block、list）；表格按表头+行+摘要单独索引。
- 代码解析升级为 AST/tree-sitter 级符号识别（function、class、enum、macro、typedef），替代当前的正则启发式。
- 同 symbol / 同章节扩展（当前只做邻近 + 文件依赖扩展）。

## 5. 表格处理

表格不能简单当普通文本。应该：

1. 保留表格标题/所在章节
2. 拆成：表头 + 每一行 + 整体摘要
3. 转成可检索文本，例如：

```
错误码 -1003 表示设备被占用，解决方案是关闭其他程序。
```

这样 BM25 能精确命中 `-1003`，向量检索也能理解"设备被占用怎么办"。

## 6. Embedding 模型选择

### 当前问题

ONNXMiniLM_L6_V2 是英文模型，对中文理解弱：

```
"相机初始化" 和 "连接相机" → 可能觉得不太相关（中文语义差）
"camera init" 和 "connect camera" → 觉得很相关（英文语义好）
```

### 模型对比

| 模型 | 大小 | 中文 | 英文 | 中英混合 | 长文本 | 速度 |
|------|------|------|------|---------|--------|------|
| MiniLM（当前） | 79MB | 差 | 好 | 差 | 256 token | 快 |
| bge-small-zh | 100MB | 好 | 一般 | 一般 | 512 token | 快 |
| **bge-m3** | **2GB** | **好** | **好** | **好** | **8192 token** | 中等 |

### 建议

中文场景选 **bge-m3**，能同时理解"相机初始化" ≈ "camera_init" ≈ "CameraOpen"。

换模型只需改 embedder.py 一个文件，但换后需重建索引。

## 7. 维度越高越好吗

不是。维度和效果是边际递减关系：

```
效果
 ↑    ___________
 |   /
 |  /
 | /
 |/
 +───────────────→ 维度
```

- 维度翻倍 → 存储、检索、计算都翻倍
- 但效果不会翻倍
- 维度太高还会导致"维度灾难"：向量之间距离差异变小，检索分不清谁更相关

对于当前场景（中文文档 + 英文代码 + SDK 术语），**bge-m3 的 1024 维是合适的平衡点**。

## 8. RAG 幻觉问题及防幻觉策略

### 问题

检索到了正确内容，LLM 还是编造了不存在的内容。

### 原因

1. LLM 倾向于"帮"你 — 宁可编一个也不说不知道
2. 训练知识干扰 — 把自研 SDK 和通用库知识混在一起
3. 检索内容不够明确 — 文档没说不支持，LLM 就脑补支持

### 已实施的防幻觉策略（三层）

**第 1 层：严格 prompt**

- 核心规则从"不要编造"改为"没有就说不存在"
- 禁止脑补、禁止用训练知识补充

**第 2 层：强制标注来源**

- 每个事实后面必须标注 `[来源: 文件路径:行号]`
- 无法标注来源的陈述不要写

**第 3 层：分离事实和推断**

回答分三块：

```
## 事实（来自文档）
- 只写检索结果里有的

## 推断（文档中未明确说明）
- AI 猜的必须标"（推断）"

## 结论
- 一句话总结
```

## 9. RAG 来源标注与上下文治理

P6 后，RAG 结果进入 workflow 上下文时必须满足：

- 每个检索片段保留 `file_path`、`start_line`、`end_line`、`source_name`。
- 回答或角色上下文中的事实应能回溯到 `context_sources` 或当前文件读取结果。
- RAG 只能补充候选事实，不能覆盖当前 workspace 中直接读取到的文件内容。
- 当 RAG 与当前文件冲突时，以当前文件事实为准，并在 trace 中记录冲突来源。
- memory 注入 Context Pack 时必须在 fact 中携带 memory id 和 source，并在 `context_sources` 中记录 scope、source、status、reason；当 memory 与当前 stage summary 冲突时，不注入 fact，只记录 `conflict_current_fact` 来源。
- memory 与 RAG 分层使用：RAG 解决项目知识定位，memory 记录用户偏好/项目决策；二者都不能存放凭证或临时调试内容。

### 数据边界

- RAG 索引默认只保存用户显式选择目录中的 chunk、来源路径和检索元数据。
- `.env`、凭证、密钥、临时调试日志和未授权外部数据不得进入 RAG 索引或长期 memory。
- 客户代码和文档默认保存在本地索引目录；如果通过模型、插件或 MCP 使用检索片段，需要在 trace 中保留 `context_sources`。
- memory 删除、更新、supersede 与审计保留策略属于 P7 治理任务，P6 先明确不能把一次性 run 状态自动沉淀为长期 memory。

### 任务结果知识回流规则

一次任务完成后，产物只能按用途进入三类位置：

- Run artifact：保存 PRD、实现说明、QA 报告、复盘、trace 和 metrics，作为本次 run 的证据与恢复材料。
- RAG 索引：只索引用户显式选择的项目文件和文档 chunk，用于后续定位事实来源。
- Long-term memory：只保存经人工确认、可跨任务复用的用户偏好、项目决策、workflow 经验或外部引用。

禁止自动回流到 long-term memory 的内容包括：一次性 run 状态、trace 片段、临时调试结论、stacktrace、未验证猜测、凭证、`.env`、API key、token、password 和 private key。复盘内容进入 memory 前必须先通过敏感内容、短期内容、scope/source 和人工确认检查；未确认的长期候选只能进入 review 审计，不得直接写入 memory store。

## 10. Agentic RAG vs 普通 RAG

### 核心区别

```
普通 RAG：检索一次，直接回答
Agentic RAG：LLM 自己决定怎么查、查几次、要不要调整策略
```

### 举例

用户问："花屏 -1003 怎么办？"

**普通 RAG**：一次检索，可能只找到错误码或只找到花屏原因

**Agentic RAG**：
1. 查"错误码 -1003" → 找到：设备被占用
2. 查"花屏原因" → 找到：驱动版本不匹配
3. 查"驱动版本兼容性" → 找到：SDK 3.2+ 需要驱动 5.1+
4. 整合三步结果给出完整答案

### 什么时候需要 Agentic RAG

- 不需要：简单明确的问题，一次检索能找到
- 需要：跨模块、跨项目、需要多次追查的复杂问题

## 10. Agentic RAG 如何判断一次检索够不够

### 三种判断方式

**方式 1：规则判断（快，便宜）**

- 检索结果为 0 → 立刻再查
- RRF 分数低于阈值 → 相关性不够，换关键词再查
- 关键词覆盖不全 → 针对缺失关键词再查
- 来源覆盖单一 → 可能不完整，再查其他文件

**方式 2：LLM 判断（慢，贵，最灵活）**

- 让 LLM 看检索结果，判断够不够回答问题
- 不够的话同时给出下一个查询建议

**方式 3：混合（推荐）**

- 先规则快速判断，明确不够的就直接再查
- 拿不准的再问 LLM（只需回答"够"或"不够"，成本很低）

### 必须设上限

最多 3 轮检索，超过后不管够不够都生成答案，标注"信息可能不完整"。

## 11. 混合检索权重怎么调

### 当前

RRF 天然平等，不设显式权重：

```
RRF 分数 = 1/(60 + 向量排名) + 1/(60 + BM25排名)
```

### 问题

用户提问类型不对称：

```
"相机怎么连"          → 向量更擅长（语义理解）
"init_camera"         → BM25 更擅长（精确匹配）
"-1003"               → BM25 远优于向量（错误码）
"MV_CC_StartGrabbing" → BM25 远优于向量（SDK 函数名）
```

### 建议：按问题类型自动调权重

```
问题包含函数名/错误码/类名？  → w_vec=0.3, w_bm25=0.7
问题全是中文模糊提问？        → w_vec=0.7, w_bm25=0.3
两者都有？                    → w_vec=0.5, w_bm25=0.5
```

判断方式：问题里含大写+下划线、0x 十六进制、纯数字+横杠 → 偏 BM25；纯中文+"怎么""如何" → 偏向量。

## 12. RAG 端到端延迟优化

### 延迟分布

```
① 查询理解 + 分词        ~0.5s
② 向量检索 + BM25 检索    ~0.5-2s
③ RRF 融合                ~0.01s
④ LLM 生成答案            ~3-15s  ← 瓶颈，占 70-90%
```

### 优化措施

| 优化项 | 效果 | 改动量 |
|--------|------|--------|
| 检索两路并行 | 延迟砍半 | 小 |
| 减少 LLM 输入量（chunk 数量 10→5，截断 1500→800） | 生成更快 | 小 |
| 流式输出 | 体感 15s→1-2s | 中 |
| 简单问题不调 LLM | 部分查询 15s→1s | 小 |
| 查询缓存 | 重复查询秒返回 | 中 |
| 增量索引 | 大项目重建快 | 中 |

### 建议

先用真实 SDK 项目跑，测量每段实际耗时，找到真正瓶颈再优化。

## 13. 索引更新策略

### 当前

全量重建：文档改一个字 → 整个索引删掉重来。

### 应该做：增量索引

按文件内容 hash 判断变化：

```
1. 读取 meta.json 中的文件 hash 记录
2. 扫描目录，计算每个文件的新 hash
3. hash 没变 → 跳过
   hash 变了 → 重新分块，删除旧 chunk，加入新 chunk
   文件删了 → 删除旧 chunk
4. 重建 BM25（BM25 不支持增量删除）
5. ChromaDB 增量删除和添加（支持）
6. 更新 meta.json 中的 hash 记录
```

## 14. Token 成本控制

### 关键事实

- **创建索引零 token** — 全程不调用 LLM
- **Token 只在查询时的 LLM 生成步骤产生**

### 控制措施

**输入 token：**

1. 减少送进 LLM 的 chunk 数量（10→5）
2. 截断每个 chunk 内容（1500→800 字符）
3. 按 RRF 分数过滤低相关 chunk
4. 压缩检索结果（去掉空行、冗余注释）

**输出 token：**

5. 限制 max_tokens（4096→1024-2048）
6. 精简 prompt

**跳过 LLM：**

7. 简单问题（函数名、错误码精确命中）直接返回检索结果

## 15. Message History 与 RAG 的职责边界

### 职责划分

| 层 | 职责 | 生命周期 | 存储 |
|---|------|---------|------|
| Message History | 当前 run 内的角色对话上下文 | 单次 run | RunStore events |
| RAG（短期） | 当前项目的代码和文档检索 | 索引存在期间 | ChromaDB + BM25 |
| 长期记忆 | 跨 run 的稳定知识和决策 | 显式管理 | 独立 knowledge store |

### 边界规则

1. **Message History 不入 RAG**：角色间的对话消息、临时推理过程、失败输出不写入长期记忆
2. **RAG 不替代 Message History**：当前 run 的上下文通过 message buffer 传递，不依赖检索
3. **长期记忆严格入库**：只有经过验证的稳定知识才写入长期记忆（见下节）

---

## 16. 长期记忆写入规则

### 入库条件（全部满足才可写入）

1. **稳定性**：信息不会在短期内变化（排除临时日志、推测、失败输出）
2. **验证性**：信息已被确认为正确（用户确认、测试通过、代码验证）
3. **复用性**：信息在未来 run 中有参考价值（排除一次性上下文）

### 入库流程

```
原始信息 → 总结提炼 → 分类标注 → 去重检查 → 写入 knowledge store
```

### 分类标签

| 类别 | 示例 |
|------|------|
| 项目决策 | "采用 PostgreSQL 而非 MySQL，因为需要 JSONB" |
| 架构约束 | "所有 API 必须经过 auth middleware" |
| 复盘结论 | "mock 测试无法覆盖数据库迁移问题" |
| 用户偏好 | "PR 合并前必须有 2 个 reviewer" |

### 禁止入库

- 临时调试日志
- 未验证的推测
- 单次 run 的中间状态
- 已过期的决策（被新决策覆盖）

---

## 17. 记忆过期和删除机制

### 过期策略

| 类型 | 默认保留期 | 触发删除 |
|------|-----------|---------|
| 项目决策 | 无限期 | 被新决策显式覆盖 |
| 架构约束 | 无限期 | 代码重构后失效 |
| 复盘结论 | 90 天 | 超期自动标记为待审核 |
| 用户偏好 | 无限期 | 用户显式撤销 |

### 删除规则

1. **显式覆盖**：新决策写入时，标记旧决策为 `superseded`
2. **超期审核**：超过保留期的记忆标记为 `pending_review`，不自动删除
3. **手动清理**：用户可通过 CLI 查看和删除过期记忆
4. **去重合并**：写入时检测语义重复，合并而非新增

### CLI 接口（Planned）

```bash
opc memory list              # 列出所有长期记忆
opc memory gc                # 清理过期和重复记忆
opc memory delete --id <id>  # 删除指定记忆
```
