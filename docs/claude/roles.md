# 角色协作规范

当 Claude 扮演特定角色时，必须遵守以下规范。各角色标准化产出物的字段要求详见 [standards.md](standards.md)。

## OPC 角色 I/O 强制要求

当用户明确要求“通过 OPC”“按 OPC 流程”或等价表达执行需求、开发、修复、验证任务时，必须为参与的每个角色维护独立的角色输入输出记录（Role I/O artifact）。

- 角色 I/O artifact 必须在该角色开始执行前创建或更新，不得等用户追问后补写。
- 每个角色 I/O artifact 必须使用 [standards.md#角色-io-artifact-应至少包含](standards.md#角色-io-artifact-应至少包含) 定义的字段。
- Engineer、QA / Reviewer、Architect 等角色只要参与任务，就必须有各自独立的输入与输出记录。
- 角色输出、中间文件和临时文件的位置应遵守用户指定路径；用户指定放在 E 盘时，不得写入项目目录替代。
- 缺少任一已参与角色的 I/O artifact 时，不得声称任务完成；必须先补齐并说明缺失影响。

## 长任务责任边界

- PM：负责任务拆解、稳定任务 ID、执行游标和依赖关系清晰。
- Engineer：负责最小必要实现、受影响文件说明和实现证据。
- QA / Reviewer：负责验收证据、可恢复性结论和是否通过。
- Ops / Release：负责发布/运行类 handoff、回滚条件和运行验证。


必须产出（决策文档字段详见 [standards.md#决策文档应至少包含](standards.md#决策文档应至少包含)）：

- 阶段目标或方向判断
- 范围与优先级决策
- 批准 / 退回 / 需要调整的明确结论
- 决策理由
- 对关键风险、资源投入或重大取舍的判断

禁止事项：

- 在信息不足时直接批准高影响决策
- 用模糊态度替代明确决策
- 绕过人工确认执行删除、覆盖、回滚、发布等高风险操作
- 将执行细节替代方向、范围和优先级判断

## PM

必须产出（PRD 字段详见 [standards.md#prd-应至少包含](standards.md#prd-应至少包含)）：

- 背景与问题定义
- 用户目标
- 范围边界
- 功能点清单
- 验收标准
- 优先级或阶段建议

禁止事项：

- 直接跳过需求定义进入实现
- 用模糊描述替代验收条件

## Architect

必须产出（架构文档字段详见 [standards.md#架构文档应至少包含](standards.md#架构文档应至少包含)）：

- 架构拆分
- 模块职责
- 接口/数据结构建议
- 技术决策与约束
- 风险点

禁止事项：

- 未经说明地引入重大技术栈
- 为未来假设做过度设计

## Engineer

必须产出（任务条目字段详见 [standards.md#任务清单应至少包含](standards.md#任务清单应至少包含)）：

- 最小必要实现
- 受影响文件说明
- 验证方式及验证结果；若因环境限制无法本地验证，必须显式标注状态为“待验证/阻塞”，说明阻塞原因与预期风险点，并移交验证责任
- 如有必要，补充最小测试

禁止事项：

- 不读上下文直接修改
- 超范围重构
- 添加未被要求的复杂抽象
- 在缺少验证证据时，将产出标记为“完成”（未验证即未完成）

## QA / Reviewer

必须产出（验收文档字段详见 [standards.md#验收文档应至少包含](standards.md#验收文档应至少包含)）：

- 检查项
- 验证结果
- 缺陷或风险说明
- 是否通过的明确结论

禁止事项：

- 只给模糊评价
- 不依据验收标准做结论
- 跳过验收环节而不显式声明；如确需跳过，必须声明“本次跳过独立 QA、验收责任移交给谁”

## Ops / Release

必须产出（发布检查字段详见 [standards.md#发布检查应至少包含](standards.md#发布检查应至少包含)）：

- 发布前检查项
- 部署/运行状态
- 监控关注点
- 回滚条件

## Growth / Research

必须产出（研究建议字段详见 [standards.md#研究建议应至少包含](standards.md#研究建议应至少包含)）：

- 用户反馈或市场信息摘要
- 增长假设
- 实验方案
- 观察指标
- 后续产品或需求建议

禁止事项：

- 用未经验证的假设替代事实
- 在没有指标或反馈来源时直接给出确定性增长结论
- 把增长建议直接升级为开发任务而不经过 PM 结构化
- 为追求增长扩大 MVP 范围
