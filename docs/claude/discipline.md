# 执行纪律

## 与 harness 方法论一致的流程

默认遵循以下顺序：

1. 明确目标
2. 结构化需求
3. 设计方案
4. 拆解任务
5. 执行实现
6. 验证结果
7. 记录结论与经验

## 长任务与跨会话执行纪律

长任务指需要多轮实现、跨文件验证、阶段验收或可能清空上下文后继续的任务。

### 上下文清空后继续协议

- 继续任务时不得依赖聊天历史，必须从任务文件和可重新读取的产物恢复上下文。
- 开始前先读取当前任务条目、`depends_on` 前置任务、`read_before_start` 指定文件、当前 pending diff、最近 `evidence` 和 `handoff`。
- 若依赖产物、验证记录或 handoff 缺失，应先记录缺失原因和影响范围，不得凭记忆补全任务状态。
- 恢复后先确认当前任务、下一任务、阻塞项、最近验证结果，再继续实现或验收。

### subagent 使用边界

- Explore subagent 只用于独立探索、文件定位、代码路径追踪和只读 review。
- 需要写文件、运行验证、勾选任务、维护 evidence、提交变更时，由主会话负责。
- subagent 不作为长期上下文保存器；它的结论必须沉淀到任务文件、规范文档、验收记录或可读取产物。
- skill 用于生成结构化结果或执行既有流程，不替代任务文件、evidence 或 handoff。

### 长任务执行检查清单

#### 开始任务前

- 读取当前任务条目和 `depends_on`。
- 读取 `read_before_start` 列出的文件、相关产物和 pending diff。
- 确认当前执行主体是 main、explore、skill、manual_review 还是 qa_acceptance。
- 识别本次只做一个最小闭环，不把定义、实现、验证和迁移混在一起。

#### 完成任务后

- 先做定向验证，再写 evidence。
- evidence 必须记录产物路径、检查方式和结果。
- 验证失败或未验证时，必须写明原因，不得只写“已完成”。
- 只在验证通过后更新任务状态或勾选完成。

#### 换会话前

- 写 handoff，包含当前状态、下一任务、阻塞项、需重读文件和最近验证结果。
- 如果任务尚未完成，补充下一步和恢复所需产物。
- 不要依赖聊天历史保留上下文。


- 一个阶段或一组依赖任务已完成并通过验收。
- 上下文过长，继续执行会影响读取关键文件或判断 pending diff。
- 即将从文档规则切换到代码实现、从实现切换到 QA、或切换到另一个模块。
- 当前任务被阻塞，需要等待人工确认或外部结果。

压缩或换会话前必须先写 handoff，至少包含当前状态、下一任务、阻塞项、需重读文件和最近验证结果。

## 人工介入节点

以下情况应优先请求用户确认：

- 方向或范围发生明显变化
- 需要在多个实现方案中做取舍
- 涉及删除、覆盖、回滚等高影响操作·
- 涉及外部系统、共享环境或不可逆动作

## MVP 优先

若当前任务存在"做完整"和"先跑通"两种路径，默认优先选择能更快形成可验证闭环的方案。

## 一致性要求

所有输出尽量统一使用以下术语：

- harness
- agent
- 角色
- 工作流
- MVP
- 验证
- 闭环

除非有明确需要，不随意引入新的同义表达替换这些核心术语。

## 对代码与实验的要求

- 优先交付可验证结果，而不是概念性描述。
- 实验性方案必须说明目标、假设、验证方法和退出条件。
- 任何自动化能力都应说明边界和失败处理方式。
- 尽量避免难以维护的复杂编排，先保证链路清晰。

## 对项目文档的要求

- 文档应服务于执行，而不是仅用于展示。
- 每份关键文档应说明它解决什么问题、供谁使用、输出什么结果。
- 文档结构应稳定，便于人类和 Agent 复用。
- 当项目方向、角色、流程发生变化时，应优先更新总纲和规范文件。

## 默认协作结论

在没有额外说明时，Claude 应默认采用以下策略：

- 以项目总纲为最高级别项目语义参考
- 以最小闭环为优先目标
- 以清晰角色分工替代混合式回答
- 以可执行文档替代理念性长文
- 以验证和反馈作为任务结束条件的一部分

## 文件维护要求

当项目进入下一阶段时，Claude 应优先检查以下内容是否需要同步更新：

- `README.md` / `tasks-beta-completion.md`：项目目标、范围、架构、路线图和阶段任务是否变化
- `CLAUDE.md`（索引至 `docs/claude/`）：协作规则、角色规范、执行纪律是否变化

如果发现文档内容与实际执行方式不一致，应优先指出并建议修正。
