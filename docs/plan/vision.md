# 项目定位与愿景

## 1. 项目定位

本项目的目标，不是单独做一个会写代码的 AI Agent，而是构建一个面向"单人软件公司"的 AI 操作系统。

所谓"单人软件公司"，指的是由一个人负责最终决策，但把产品、架构、研发、测试、运维、增长等职责拆分为可协作的角色，并通过 Agent、工具链与评估闭环，把这些角色组织成一个可重复运行的工程系统。

本项目选择 **Harness Engineering** 作为核心方法论，不把 AI 视为单次生成工具，而把它放入一个可计划、可执行、可验证、可干预、可持续优化的运行框架中。最终希望形成一套既能支持个人开发，又能逐步扩展为多 Agent 协同的软件生产体系。

## 2. 为什么选择 Harness Engineering

仅靠单个大模型或单个 Agent，很难稳定解决真实软件工程中的复杂问题。问题通常不在"模型不会写"，而在于：

- 需求缺乏结构化表达
- 多角色职责混杂，缺少稳定分工
- 工具调用、权限控制、上下文管理不统一
- 生成结果缺少验证与回路
- 缺少人工介入点，难以把控质量与风险

Harness Engineering 的价值在于：为 Agent 系统提供一整套工程约束，包括任务拆解、角色分工、执行环境、权限边界、记忆机制、验证流程和人工审查节点。

因此，本项目的重点不是训练一个更"聪明"的 Agent，而是构建一个更"可靠"的 Agent 公司系统。

## 3. 能力边界

OPC 按严格标准区分能力状态：只有同时具备代码实现、测试覆盖、可本地复现路径的能力，才标记为当前能力；尚未满足该标准的能力，不作为已实现能力对外承诺。

### 3.1 当前能力

当前能力聚焦软件交付编排的最小闭环：

- PM、Engineer、QA 等角色的职责定义与结构化输出协议
- PM → Engineer → QA 的最小工作流流转、失败回流与轮次限制
- workflow state、role outputs、events、metrics 和 run trace 的基础记录
- 基于 BM25 的本地知识检索与查询入口
- 面向文档、代码和命令执行的基础工具能力

### 3.2 实验能力

实验能力已有部分实现或设计，但尚未满足“有代码、有测试、可本地复现”的完整当前能力标准：

- Agent 消息缓冲区、工具调用与工具结果的统一持久化
- 中断后的完整状态恢复入口
- 工具注册协议、工具审计与更细粒度权限边界
- 工作流质量指标、成本预算观测与历史 trace 展示
- 符号级代码检索、import graph 和测试文件关联

### 3.3 长期愿景

长期愿景是把 OPC 从最小软件交付编排层扩展为更完整的单人软件公司运行框架：

- 支持自定义 workflow、并行任务和子工作流
- 支持更完整的角色组合、项目记忆和复盘沉淀
- 支持多项目、多环境和更系统化的发布运维闭环
- 逐步接近“单人软件公司操作系统”的远期形态

## 4. 项目目标与非目标

### 4.1 项目目标

1. 建立适用于单人团队的 AI 软件交付编排框架。
2. 将产品、技术、测试、运维等职责抽象为可协作的 Agent 体系。
3. 先形成从需求到实现到验证的最小闭环，再逐步扩展到上线与运营。
4. 建立基于 harness 的任务拆解、执行、验证、反馈机制。
5. 在 MVP 阶段落地一个可运行、可验证、可迭代的最小系统。

### 4.2 非目标

当前阶段不以以下事项为主要目标：

- 不追求做通用型 AGI 平台
- 不追求一次性覆盖所有企业管理能力
- 不优先做复杂的多租户平台化产品
- 不以"炫技式多 Agent 编排"替代可落地的工程闭环
- 不在 MVP 阶段过早引入过重基础设施

## 5. 项目核心定义

### 5.1 项目本质

本项目可以定义为：

> 一个以 Harness Engineering 为方法论、以 Agent 角色体系为组织方式、以软件交付闭环为目标的单人软件公司操作系统。

### 5.2 核心设计原则

- **角色分工优先**：先定义职责，再定义模型与工具。
- **流程闭环优先**：先保证需求到交付可验证，再考虑规模化。
- **人工监督优先**：关键节点允许人类确认、拒绝、回滚。
- **小步迭代优先**：先做最小链路，再逐步扩展角色与能力。
- **文档驱动优先**：角色协作建立在明确输入输出之上。
- **可观测优先**：所有关键执行过程都应可追踪、可复盘。
