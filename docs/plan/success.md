# 交付标准与成功标准

## 1. 本项目的交付标准

一个合格的项目阶段性交付，至少应满足：

- 文档结构清晰，可被人和 Agent 共同理解
- 角色分工明确，不依赖隐含知识协作
- 工作流可执行，而非仅停留在概念层
- MVP 范围收敛，有明确阶段边界
- 能为后续实现提供直接操作指引

## 2. 项目愿景与成功标准

### 2.1 项目愿景

本项目的愿景，是构建一个由单人掌控、由多角色 agent 协作推进、由 harness 机制保障执行质量的软件公司操作系统。

它的目标不是替代人的判断，而是把人的角色从"亲自完成每一步"提升为"负责定义方向、审批关键决策、处理例外情况"。

如果项目推进顺利，最终应形成一套可复用的方法与工作系统，使单人开发者也能以接近小型组织的方式运行产品研发、交付、验证与优化流程。

### 2.2 项目期望

项目的合理期望应分阶段理解：

#### L1：方法论与文档层成功

达到以下状态即可视为第一层成功：

- 项目总纲清晰
- 角色体系清晰
- 协作规则清晰
- 文档模板可复用
- Claude / agent 的执行边界明确

#### L2：MVP 层成功

达到以下状态即可视为第二层成功：

- 至少跑通 1 次完整任务闭环
- 该闭环包含真实需求、真实实现、真实验证、真实复盘
- 至少 3 个角色可以稳定协作
- 有最小工具执行环境与最小权限边界
- 能发现并修正工作流中的明显问题

#### L3：系统层成功

达到以下状态即可视为第三层成功：

- 能持续完成多类软件任务，而非一次性演示
- 角色交接稳定，输入输出规范可重复使用
- 验证、回退、复盘机制开始稳定发挥作用
- 文档、记忆、流程可以随着任务执行持续优化
- 系统开始具备个人软件公司操作台的雏形

### 2.3 这个项目最终可以做到什么程度

在理想情况下，本项目可以成长为：

- 一个 AI 驱动的软件生产系统
- 一个单人创业者的工程组织层
- 一个可复用的多角色 agent 公司框架
- 一套以 harness 为核心的软件交付方法体系

在现实可落地的层面，更重要的价值在于：

- 把软件开发流程标准化
- 把 AI 从工具升级为角色化协作者
- 显著减少需求整理、任务拆解、文档同步、验证复盘上的重复劳动
- 形成适合个人长期复用的软件开发操作系统

### 2.4 当前阶段做不到什么

为了保持预期清晰，当前阶段不应假设系统能够：

- 完全替代人的产品判断
- 完全自主做长期商业决策
- 在复杂开放环境中稳定全自动创业
- 低成本、高准确率地处理所有软件任务
- 长时间脱离人工监督稳定运行

因此，本项目的目标应理解为：

> 不是构建"无人公司"，而是构建"以人为核心决策者、以 agent 为角色团队、以 harness 为治理机制"的单人软件公司系统。

### 2.5 阶段判断依据

#### Phase 1 完成判断

- 项目总纲、协作规范、核心模板已具备
- 至少一条最小工作流已定义清楚
- 能选出一个适合演示的真实任务

#### Phase 2 完成判断

- 至少一次端到端闭环已成功跑通
- 角色交接中出现的问题已被记录并修正
- 文档、任务、验证之间已建立稳定关系

#### Phase 3 完成判断

- 已具备基本发布、回滚、观测与复盘能力
- 系统可以连续处理多个任务而不是单次试验
- 失败案例能反向推动流程优化

#### Phase 4 完成判断

- 系统可复用于更多项目类型
- 角色和模板具备较强复用性
- 形成较稳定的单人软件公司运行框架

## 3. 工业化评测基准

P6 后的工业化评测不以“演示能跑一次”为目标，而是用固定 case 比较后续版本的稳定性。第一版基准集见 [Industrial Benchmark Cases](../runs/industrial_benchmark_cases.md)，覆盖：

| Case | 评测目的 |
| --- | --- |
| bugfix | 验证缺陷复现、最小修复、定向测试和 QA 回流 |
| review | 验证只读评审输出是否可行动且不会写入 |
| docs-update | 验证文档漂移修复和链接/索引检查 |
| config-drift | 验证配置漂移能被 `opc config validate` 发现 |
| failed-tool | 验证工具失败或危险操作能产生可定位 trace |
| rework-resume | 验证 QA fail 后返工与 resume 不丢失上下文 |

统一指标为任务成功率、人工介入率、成本、耗时和失败可定位性。评测结果必须引用固定输入、预期产物、验收标准、trace 证据和失败判定，避免把一次成功 walkthrough 当作工业化稳定性证明。

验收日期：2026-05-21。P6 “Harness 使用层与工业化落地”达到单人本地工业落地最小闭环，验收结论为通过。

| 验收项 | 证据 |

| Harness L1-L6 | 架构文档已说明信息边界、工具系统、执行编排、记忆状态、观测评估、约束恢复的现状与不做项 |

P6 验收采用定向回归而非全量重跑，避免本机性能风险；后续版本比较应继续使用工业化评测基准的固定 case 与 trace 证据。

## 5. 结论

本项目的长期目标，是把 AI 从"辅助写代码的工具"升级为"可组织、可约束、可验证、可持续运行的软件公司系统"。

其核心不是堆叠更多 Agent，而是建立一套以 harness 为核心的工程秩序：谁来做、按什么标准做、做完如何验证、失败如何回退、经验如何积累。

如果这套系统能够稳定运行，那么单人开发者就不再只是"一个人在写代码"，而是在经营一家由自己掌控、由 Agent 协作推进的软件公司。
