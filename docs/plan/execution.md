# 执行计划与模板

## 1. 第一阶段待落地事项

1. 完成项目主计划书与项目级协作规范文档。
2. 固化角色清单及每个角色的输入输出定义。
3. 建立最小文档模板：PRD、架构说明、任务清单、验收标准。
4. 选定最小工具链与执行方式。
5. 设计一个演示型任务，验证 PM → Engineer → QA 的最小闭环。
6. 记录第一次执行中的问题，反推需要补充的 harness 机制。

## 2. 项目执行 Todo List

### 2.1 当前优先级

1. **梳理项目愿景与成功标准**
   - 明确项目目标、边界、成功标准与阶段判断依据。
2. **定义角色体系与协作协议**
   - 固化 CEO、PM、Architect、Engineer、QA、Ops、Growth 等角色的职责、输入输出、交接规则与审批节点。
3. **建立核心文档模板**
   - 设计 PRD、架构说明、任务清单、验收标准、复盘记录等模板。
4. **设计最小 harness 工作流**
   - 明确需求 → 方案 → 任务 → 实现 → 验证 → 复盘的最小闭环。
5. **确定执行基础设施选型**
   - 确定 MVP 所需的文档、代码、终端、记忆、任务跟踪与权限控制方案。
6. **落地首个演示任务闭环**
   - 选择一个足够小但完整的软件任务，真实跑通 PM → Engineer → QA 的闭环。
7. **建立验证与复盘机制**
   - 建立验收、失败记录、问题归因和经验沉淀机制。
8. **规划 Phase 2-4 扩展路线**
   - 在 MVP 跑通后，规划架构增强、运维观测、增长角色与长期记忆等后续阶段。

### 2.2 完成顺序建议

- **阶段 A：定义规则**
  - 梳理项目愿景与成功标准
  - 定义角色体系与协作协议
  - 建立核心文档模板
- **阶段 B：跑通闭环**
  - 设计最小 harness 工作流
  - 确定执行基础设施选型
  - 落地首个演示任务闭环
- **阶段 C：增强系统**
  - 建立验证与复盘机制
  - 规划 Phase 2-4 扩展路线

## 4. MVP 执行基础设施选型

MVP 阶段不追求完整平台化，而是选择最小但足够稳定的执行基础设施，重点保证文档驱动、任务可追踪、修改可验证、风险可控制。

### 文档层

- 项目总纲：`plan.md`（索引至 `docs/plan/`）
- 协作规范：`CLAUDE.md`（索引至 `docs/claude/`）
- 任务产物：使用 PRD、架构说明、任务清单、验收记录、复盘记录模板
- 原则：所有关键决策必须落文档，不依赖聊天上下文作为唯一信息源

### 执行层

- 文档编辑：Claude 直接维护 Markdown 文档
- 代码修改：Claude 通过受控编辑方式修改项目文件
- 搜索与阅读：优先基于文件读取和内容搜索理解上下文
- 终端执行：仅在需要验证、运行命令或检查环境时使用

### 任务层

- 使用任务列表管理 todo 与执行状态
- 每个任务至少有：目标、责任角色、状态、完成标准
- 复杂任务必须拆成多个可验证子任务

### 记忆层

- 长期稳定约束写入 `plan.md` 与 `CLAUDE.md`
- 可跨任务复用的经验写入 memory
- 当前轮临时上下文不作为长期规范保存

### 验证层

- 文档任务：检查结构完整性、术语一致性、上下游是否可直接使用
- 实现任务：至少给出最小验证方式，如测试、运行结果、手工检查路径
- 验收任务：必须形成明确通过/退回结论

### 权限与风险控制

- 高影响操作默认需要人工确认
- 删除、覆盖、回滚、外部系统操作默认视为高风险
- 范围变化、技术大取舍、发布决策需要升级确认

## 5. Harness Engineering 执行规则

在执行层，Harness Engineering 体现为每个任务都必须满足的最小约束，而不是额外的理论说明。

| 约束 | 执行要求 | 验证方式 |
| --- | --- | --- |
| 可执行 | 任务进入 Engineer 前必须有目标、范围、输入和完成标准 | 检查 PM 输出或任务清单是否包含这些字段 |
| 可验证 | 每个实现或文档改动都要给出最小验证路径 | 运行定向测试、命令 dry-run，或做结构化文本检查 |
| 可恢复 | 关键状态、产物路径、失败原因应写入 state、trace 或任务记录 | 检查 artifacts、run trace 或任务清单状态 |
| 可观测 | Agent 输入输出、耗时、token、工具调用和错误应可追踪 | 检查 run metrics、events 或审计记录 |
| 可干预 | 范围变化、高风险命令、验收争议要升级给用户 | 检查是否存在人工确认点或 next_action |

与 SDLC、Agile、DevOps 的关系：OPC 不用 Harness Engineering 替代这些实践，而是把它们在 AI Agent 执行中的关键控制点具体化。SDLC 提供阶段语义，Agile 提供小步反馈原则，DevOps 提供运行和观测要求；Harness Engineering 负责把这些要求落实为 Agent 可遵守的输入输出、状态流转、验证和审计规则。

### MVP 推荐工具组合

| 层级 | MVP 选择 | 目的 |
| --- | --- | --- |
| 文档 | Markdown + 项目主文档 | 统一知识与规则 |
| 任务 | 内置任务列表 | 跟踪进度与责任 |
| 编辑 | 受控文件读写 | 保证修改可追溯 |
| 验证 | 最小测试 / 手工检查 | 保证结果不是空想 |
| 记忆 | 项目文档 + memory | 沉淀稳定经验 |
| 审批 | 人工确认节点 | 控制高风险行为 |

### 当前阶段不引入的能力

- 复杂工作流编排平台
- 大规模并发 agent orchestration
- 重型监控与观测平台
- 复杂向量记忆检索系统
- 自动化商业增长系统

## 5. 首个演示任务闭环设计

为了验证本项目不是"只有总纲，没有运行能力"，MVP 必须跑通一个真实的小任务闭环。

### 演示任务选择标准

首个任务应满足：

- 足够小，避免陷入长周期实现
- 足够完整，能覆盖需求、实现、验证、复盘
- 尽量贴近真实软件工程，而不是纯概念任务
- 最好能在当前项目或一个简单示例中完成

### 推荐任务类型

优先从以下任务里选择一个：

1. 为现有小模块补一个功能说明并实现对应变更
2. 新增一个简单页面或配置能力
3. 新增一个简单接口或命令行能力
4. 把一段模糊需求转为 PRD，再完成最小实现与验收

### 不推荐的首任务

- 完整产品从零搭建
- 多系统联调
- 复杂部署与运维任务
- 高不确定性的研究型任务

### 演示闭环运行步骤

1. **PM**：产出一份 PRD
2. **Architect / PM**：决定是否需要架构说明
3. **Planner**：形成任务清单
4. **Engineer**：完成最小实现
5. **QA**：基于验收标准检查结果
6. **PM / QA / 用户**：完成复盘记录

### 演示闭环完成标准

演示任务完成，不以"功能很大"为标准，而以以下事项为标准：

- 五类文档模板至少使用其中四类
- 工作流实际走完一轮
- 至少一次真实修改或真实交付发生
- 至少一次明确验收结论产生
- 至少一个流程问题被记录到复盘中

### 演示闭环建议产出

- 一份 PRD
- 一份架构说明或简化方案
- 一份任务清单
- 一份验收记录
- 一份复盘记录

## 6. 验证与复盘机制

项目不能只追求"做完"，必须建立"做完之后如何判断是否有效"的机制。

### 验证机制

每轮任务至少从三个维度验证：

1. **需求验证**
   - 是否满足 PRD 与范围约束
   - 是否超出了未授权范围

2. **实现验证**
   - 是否有最小可证据化结果
   - 是否给出测试、运行结果或手工检查路径

3. **流程验证**
   - 角色交接是否清楚
   - 输入输出是否足够支撑下游工作
   - 是否出现了需要补规范的环节

### 复盘机制

每轮结束都应回答：

- 本轮目标是否达成
- 最大阻塞点是什么
- 问题出现在需求、方案、实现还是验收
- 哪些经验值得沉淀为模板、规范或记忆
- 下次如何减少同类问题

### 失败记录要求

若任务失败，不应只记录"失败了"，而要记录：

- 失败位置
- 失败原因
- 影响范围
- 临时修复方式
- 长期修正建议

### 经验沉淀规则

以下内容应优先沉淀：

- 可重复出现的问题模式
- 有效的验收清单
- 常见回退场景
- 有用的任务拆解方式
- 被验证有效的角色协作规则

## 7. Phase 2-4 扩展路线

在 MVP 跑通之后，项目进入扩展阶段，目标不再只是"能跑一次"，而是"能稳定跑很多次，并逐步形成单人软件公司框架"。

### Phase 2：增强角色协作

目标：让多角色协作从"可运行"变成"更稳定"。

重点扩展：

- 强化 Architect 角色的常态化介入
- 细化任务输入输出协议
- 增加回退场景模板
- 形成更稳定的审批与升级机制

阶段结果应达到：

- 不同角色的职责边界更少冲突
- 文档模板可以覆盖更多任务类型
- 任务交接成本下降

### Phase 3：引入运维与观测

目标：让系统具备运行视角，而不只是开发视角。

重点扩展：

- 建立发布前检查清单
- 定义最小运行指标
- 记录执行日志、失败案例与回滚原因
- 建立简单的观测与问题归因机制

阶段结果应达到：

- 可以判断任务是否真的在运行层成功
- 可以跟踪失败而不是只看到结果
- 可以复盘发布与运行阶段的问题

### Phase 4：形成可扩展公司框架

目标：把单次项目实践提升为可复用框架。

重点扩展：

- 引入 Growth / Research 角色
- 建立跨任务可复用的记忆体系
- 提炼角色模板、任务模板、验收模板
- 支持更多任务类型和项目类型

阶段结果应达到：

- 项目具备更强迁移性
- 同一套方法可以服务多个实际项目
- 单人软件公司的运行方式趋于稳定

## 8. 首轮演示案例

以下案例作为本项目第一轮真实闭环样板，用于验证文档、角色、工作流与验收机制是否可以协同运行。

### 8.1 PRD

```md
# PRD

## 1. 背景
- 当前问题：
  - 项目已经具备总纲、协作规范、角色体系、工作流、模板和路线图，但还没有完成一次"真实跑通"的首轮实例。
  - 如果没有一个实际演示闭环，这个项目仍然停留在方法论和文档设计层，无法证明这套 harness 体系是否真的可执行。
- 为什么现在做：
  - 现在最需要的不是继续扩展抽象设计，而是验证现有规则能否支撑一次最小真实闭环。
  - 首轮演示闭环将作为后续迭代的样板，帮助发现模板缺口、角色交接问题和验证机制不足。

## 2. 目标
- 业务目标：
  - 用最小成本验证本项目的 harness 方法是否可落地。
  - 为后续真实软件任务建立一个可复制的执行样板。
- 用户目标：
  - 让我看到这套系统不是只有计划书，而是真的能按 PM → Planner / Engineer → QA → 复盘的方式跑通一次。
  - 让我得到一套可直接复用的首轮实例文档。

## 3. 用户场景
- 场景 1：
  - 作为项目发起者，我希望先跑一次小而完整的闭环，确认这套项目方法可执行，再决定是否进入更复杂的真实软件任务。
- 场景 2：
  - 作为后续使用者，我希望未来遇到新任务时，可以直接参考这次首轮样板来复用流程、模板和验收方式。

## 4. 功能范围
### 4.1 包含范围
- 为"首轮演示任务"产出一份完整 PRD
- 基于该 PRD 继续产出：
  - 任务清单
  - 必要时的简化架构说明
  - 验收记录
  - 复盘记录
- 该任务应以文档驱动为主，允许包含最小真实修改或最小真实交付
- 验证重点放在：
  - 工作流是否走通
  - 角色交接是否清晰
  - 模板是否够用
  - 验收是否可执行

### 4.2 不包含范围
- 不要求首轮就构建完整软件产品
- 不要求引入复杂基础设施、部署系统或大规模 agent 协同
- 不要求在首轮任务里验证商业增长、长期运营或多项目复用能力
- 不以"大功能实现量"作为首轮成功标准

## 5. 验收标准
- [ ] 已产出一份结构完整的首轮演示任务 PRD
- [ ] 已基于该 PRD 产出至少一份任务清单
- [ ] 已明确本轮是否需要单独架构说明
- [ ] 各文档之间上下游关系清晰
- [ ] 当前交付内容能够支撑进入下一步执行
- [ ] 至少能证明这套 harness 工作流已经开始真实运转

## 6. 风险与依赖
- 风险：
  - 任务选得过大，导致首轮闭环周期过长
  - 任务选得过小，无法真正验证角色协作与验收流程
  - 文档产出完整，但缺少真实执行感，导致验证价值不足
- 依赖：
  - 已完成的 `plan.md`
  - 已完成的 `claude.md`
  - 已定义的角色协作协议
  - 已定义的最小 harness 工作流
  - 已定义的 PRD / 任务 / 验收 / 复盘模板
```

### 8.3 架构结论

```md
# 架构结论

## 1. 结论
本轮"首个演示任务闭环"无需单独编写完整架构说明文档。

## 2. 原因
- 本轮任务目标是验证 harness 工作流是否能跑通，而不是验证复杂系统设计能力。
- 任务主体是文档驱动闭环，复杂度低，不涉及新的系统模块边界、复杂接口设计或数据结构设计。
- 当前已有 `plan.md` 和 `claude.md` 作为全局约束，已足够支撑本轮最小执行。
- 若额外引入完整架构文档，会增加流程成本，不符合 MVP 阶段"小步验证、先跑通再扩展"的原则。

## 3. 本轮采用的替代方式
本轮使用以下材料替代完整架构说明：
- PRD
- 任务清单
- 验收记录
- 复盘记录

## 4. 何时需要单独架构说明
当后续任务出现以下情况时，应恢复完整架构说明：
- 涉及多个模块协作
- 涉及明确接口设计
- 涉及数据结构或状态流转设计
- 涉及部署、运行、回滚等系统级决策
- 涉及较高实现复杂度或明显技术取舍

## 5. 本轮执行决策
因此，本轮任务按以下路径继续推进：

**PRD → 任务清单 → 最小真实交付 → 验收记录 → 复盘记录**
```

```md
## 1. 交付目标
本轮最小真实交付的目标，不是实现一个大型功能，而是完成一次真实的、可验证的 harness 闭环交付。

## 2. 本轮交付内容

- 一份 PRD
- 一份任务清单
- 一份架构结论
- 一份验收记录草案
- 一份复盘记录草案

## 3. 为什么这算"真实交付"
虽然本轮以文档为主，但它不是空泛规划，而是已经实际完成了：

- 一次真实需求结构化
- 一次真实任务拆解
- 一次真实架构判断
- 一次真实验收判断
- 一次真实复盘沉淀

本轮交付证明了以下事项：

- 项目现有模板是可用的
- 角色分工可以落地
- 工作流不是抽象描述，而是可以顺序执行

- 不追求复杂功能实现
- 不覆盖部署与运行层验证
```

```md
- 已完成最小真实交付说明
- 当前已经形成一次完整的文档驱动闭环样板

## 3. 做得好的地方
- 项目从纯方法论设计成功进入了首轮真实执行
- 各文档产物之间上下游关系清晰
- 首轮任务范围控制得较好，没有陷入过度设计
- 验收与复盘没有被省略，而是被纳入正式流程

## 4. 出现的问题
- 本轮交付偏文档型，真实"工程执行感"仍然偏弱
- 首轮虽然完成了闭环，但还没有绑定一个更具体的软件改动场景
- 架构结论被简化是合理的，但也说明本轮还未充分验证 Architect 角色在复杂任务中的价值

- 项目当前仍处于 MVP 起步阶段，优先验证流程而非复杂功能
- 当前目录下缺少现成代码型目标，因此首轮选择文档驱动闭环更稳妥
- 为了降低失败风险，首轮主动选择了低复杂度任务

## 6. 后续改进
- 下一轮应选择一个更接近真实软件开发的小任务
- 下一轮最好包含至少一次明确的文件修改或功能变更
- 可以在下一轮中正式引入 Architect 或 Engineer 的更强参与感
- 验收记录可增加更明确的"证据链接/文件位置"字段

## 7. 应沉淀到系统中的经验
- 首轮闭环任务应优先选择"小而完整"的目标
- 文档驱动可以作为 harness 首轮验证的有效起点
- 若任务复杂度低，可以用"架构结论"替代完整架构说明
- 验收与复盘必须从第一轮就纳入流程，否则系统会退化成只会产出文档的助手
```
