# review

对 pending diff、PR、任务结果或指定文件做只读评审，识别阻塞问题、非阻塞建议和风险结论；该 skill 只输出评审意见，不直接修改代码或执行修复。

## 用法

`/review <pending diff、PR、任务结果或文件范围>`

## 目标

- 找出会影响正确性、安全性、可维护性或验收结论的阻塞问题
- 区分 blocking 与 non-blocking 建议，避免把风格偏好当作阻塞项
- 用文件路径、行号、命令或 diff 片段支撑每条结论
- 给出 `pass` / `needs-work` / `needs-info` 的明确评审结论

## 适用场景

- 提交前需要检查 pending diff 是否满足任务目标
- PR 或补丁需要独立只读评审
- 任务结果需要核对范围、风险和验证证据
- 指定文件需要检查潜在缺陷或文档漂移

## 不适用场景

- 直接修改代码或文档
- 代替 `/bugfix` 执行修复
- 代替 `/acceptance-check` 给出最终 QA 验收
- 在没有 diff、文件或任务结果的情况下猜测问题

## 输出分类规则

### Blocking 问题

满足任一条件时列为 blocking，并使结论变为 `needs-work`：

- 破坏任务验收标准、核心功能或已有回归测试
- 引入安全风险、数据损坏、权限越界或高风险副作用
- 缺少必要验证，导致关键路径无法判断是否可用
- 改动范围明显超出任务目标，且可能影响无关模块
- 文档、配置或入口变更会误导用户执行危险操作

每条 blocking 必须包含：问题描述、影响范围、证据引用、建议修正方向。

### Non-blocking 建议

满足以下条件时列为 non-blocking，不单独阻止通过：

- 命名、排版、重复代码或文档表达可改进，但不影响验收
- 可读性、可维护性或测试覆盖可以增强，但当前实现仍满足目标
- 后续优化项不属于本次任务范围
- 风险较低且有明确原因说明可延后处理

每条 non-blocking 必须说明为什么不是 blocking。

### 风险判断

至少覆盖以下维度，并为每项给出 `未发现 / 有风险 / 信息不足`：

- 正确性：逻辑、边界条件、状态流转是否匹配任务
- 安全：输入、权限、命令、文件路径、敏感信息是否安全
- 兼容性：旧配置、旧 artifacts、旧入口或用户工作流是否受影响
- 测试：验证命令、测试数据和证据是否覆盖关键路径
- 发布：是否涉及部署、迁移、回滚或外部系统副作用

### 结论规则

- `pass`：未发现 blocking，且信息足以判断目标满足。
- `needs-work`：存在至少一个 blocking，必须先修正再进入下一阶段。
- `needs-info`：缺少任务目标、diff、文件内容或验证证据，无法可靠判断。

## 执行规则

1. 先确认评审对象和任务目标；缺少目标时只做通用风险评审并标记 `needs-info`。
2. 只读评审，不写入文件、不自动修复、不触发发布或外部副作用。
3. 每条 blocking 问题必须说明影响、证据和建议修正方向。
4. non-blocking 建议不得阻止通过，除非它会破坏验收标准或安全边界。
5. 评审结论必须与问题分类一致：存在 blocking 时为 `needs-work`；信息不足时为 `needs-info`；无 blocking 时可为 `pass`。

## 输出骨架

```
[评审对象] ...
[任务目标/验收标准] ...
[Blocking 问题]
- 问题 / 影响 / 证据 / 建议修正方向
[Non-blocking 建议]
- 建议 / 理由 / 证据
[风险判断]
- 正确性 / 安全 / 兼容性 / 测试 / 发布
[结论] pass / needs-work / needs-info
[后续动作] ...
```

## 验收

- 明确 review 是只读评审，不直接修改代码
- 输入覆盖 pending diff、PR、指定文件和任务结果
- 输出包含 blocking、non-blocking、风险判断和明确结论
- 每条问题或建议都要求引用文件路径、行号、命令或 diff 证据
