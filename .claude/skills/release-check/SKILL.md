# release-check

对实现结果、QA 验收、运行环境和发布对象做发布前只读检查，输出发布建议、风险和回滚条件；该 skill 只给出发布建议，不执行真实发布、部署、推送或外部系统变更。

## 用法

`/release-check <发布对象、实现结果、QA 证据或环境信息>`

## 目标

- 汇总发布对象、验证证据和环境前置条件
- 检查发布前风险、监控关注点和回滚条件是否明确
- 给出 `ready` / `needs-info` / `not-ready` 的发布建议
- 明确哪些动作需要人工确认或单独发布命令执行

## 适用场景

- 代码、文档、配置或脚本准备发布前需要最终检查
- QA 已给出结论，需要确认运行验证和回滚条件
- 变更涉及环境、部署、监控、上传或外部系统副作用
- 发布前需要整理人工确认清单

## 不适用场景

- 执行真实发布、部署、推送、上传或回滚
- 绕过 QA 验收或替代 `/acceptance-check`
- 对没有发布对象或环境信息的任务强行给出 ready
- 修改代码、配置、CI/CD 或基础设施

## 执行规则

1. 只读取用户提供的实现结果、QA 证据、环境信息和发布对象。
2. 不执行发布命令，不推送代码，不修改远端系统，不创建外部副作用。
3. 缺少 QA 通过证据、运行验证方式或回滚条件时，结论不得为 `ready`。
4. 发布对象涉及危险操作时，必须列出人工确认节点。
5. 输出必须区分“检查建议”和“真实发布动作”，避免用户误以为已发布。

## 输出骨架

```
[发布对象] ...
[输入证据]
- 实现结果：...
- QA / 验收：...
- 环境信息：...
[发布前检查项]
- ...
[运行验证方式]
- ...
[监控关注点]
- ...
[回滚条件]
- ...
[人工确认节点]
- ...
[结论] ready / needs-info / not-ready
[说明] 仅发布建议，未执行真实发布
```

## 验收

- 明确 release-check 只做发布建议，不执行真实发布
- 输入覆盖实现结果、验收结果、运行环境和发布对象
- 输出包含发布前检查项、运行验证方式、监控关注点、回滚条件和发布结论
- 缺少关键证据时不得输出 `ready`
