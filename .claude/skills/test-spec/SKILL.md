# test-spec

把 PRD、bug 描述或任务条目转成可执行、可验收的测试点，帮助 QA/Engineer 在实现前明确正常路径、失败路径和回归范围。

## 用法

`/test-spec <PRD、bug 或任务描述>`

## 目标

- 从输入中提取验收标准和风险点
- 生成分层测试点，覆盖正常路径、异常路径和边界条件
- 给出适合小步验证的命令或检查模板
- 明确哪些测试需要人工检查、哪些可以自动化

## 适用场景

- PRD 或任务需要转成 QA 测试点
- bugfix 需要补充回归测试范围
- 实现前需要确认验收路径和边界条件
- 已有验收标准需要拆成可执行检查项

## 不适用场景

- 直接修复代码缺陷
- 替代 QA 验收结论
- 在需求不清时擅自扩大测试范围
- 执行长时间、全量或高风险命令

## 执行规则

1. 先定位输入类型：PRD、bug、任务条目或验收标准。
2. 测试点必须对应可观察行为或可检查产物。
3. 缺少预期行为时先标记待澄清，不得猜测业务目标。
4. 优先推荐定向测试、文档检查或最小复现命令。
5. 输出必须区分自动验证和人工检查。

## 输出骨架

```
[输入类型] PRD / bug / task / acceptance
[测试目标] ...
[测试点]
- 正常路径：...
- 失败路径：...
- 边界条件：...
[验证命令模板]
- ...
[人工检查项]
- ...
[回归风险] ...
```

## 验收

- 说明该 skill 把 PRD、bug 或任务描述转成测试点
- 明确 QA/Engineer 责任边界
- 输出测试点可关联到验收标准或缺陷复现路径
- 验证建议优先定向、可复现、低成本
